# 顶部导航菜单复刻实施计划

> 目标：将当前 Photo-Wall 顶部胶囊导航升级为接近 `awwwards.com/websites/gsap/` 的顶部导航体验：桌面端为浅色大面板 mega menu，移动端为汉堡展开的纵向导航。本文为实施前方案文档，不记录执行结果。

## 1. 结论

当前原计划「模型圆环菜单」不再适合作为主方案。

原因：

- Awwwards 目标页面不是底部圆环菜单，而是顶部导航 + 下拉大面板。
- 桌面端核心效果是 `Explore` 触发浅灰大面板，页面内容被暗色遮罩压低。
- 移动端核心效果是左上汉堡展开固定顶部纵向菜单。
- 该效果作用于全站导航，不是底部模型切换入口。

因此本次调整为：

```text
复刻对象：Awwwards 风格顶部导航菜单
保留范围：当前 Footer 模型切换先不改成圆环
实施重点：Header 桌面 mega menu + 移动端汉堡菜单
```

### 2.1 桌面端

Awwwards 桌面端菜单主要特征：

```text
顶部固定导航栏
左侧品牌标识
主导航含 Explore / Directory / Academy / Jobs / Market
中间搜索框
右侧 Log in / Sign Up / Be Pro / Submit Website

页面主体被半透明暗色遮罩压低
面板内左侧为分组入口，右侧为当前分组的链接列表
面板有圆角、浅灰背景和清晰分隔
```

对 Photo-Wall 的适配：

```text
保留 PHOTO WALL 品牌
主导航替换为：探索 / 主题目录 / 作品视图 / 关于
搜索框第一阶段可不接入真实搜索
右侧按钮替换为：当前主题 / 提交图片 或 设置
Explore 面板承载主题、视图、项目说明等入口
```

### 2.2 移动端

Awwwards 移动端菜单主要特征：

```text
左上汉堡按钮
顶部仍保留品牌与搜索框
点击汉堡后出现固定纵向菜单
菜单占据首屏大部分高度
每一行有大行高和分隔线
可展开项右侧带箭头
底部保留社交或辅助入口
```

对 Photo-Wall 的适配：

```text
移动端显示汉堡 + PHOTO WALL + 简化搜索入口
菜单项为：探索、主题目录、塔式、纸片、关于、设置
可展开项第一版只做视觉箭头，不做多级深层菜单
底部辅助入口第一版可省略，避免过度实现
```

## 4. 需求边界

### 4.1 第一阶段目标

第一阶段只做导航效果复刻，不做完整 Awwwards 站点功能。

必须完成：

- 桌面端顶部导航结构升级；
- `Explore` 点击展开 mega menu；
- 展开时页面内容暗化；
- 点击遮罩关闭；
- 按 Esc 关闭；
- 点击导航项时复用现有 `onNavigate`；
- 移动端汉堡菜单展开；
- 移动端菜单具备大行高、分隔线、箭头视觉；
- 支持 `prefers-reduced-motion`。

暂不做：

- 真实搜索；
- 登录、注册、会员体系；
- 多层级复杂分类筛选；
- Awwwards 完整数据结构；
- 替换 Footer 模型切换；
- 新增不存在的业务页面。

### 4.2 路由映射

第一阶段导航项建议：

| 文案 | 路由 | 行为 |
|------|------|------|
| 探索 | 无直接路由 | 打开 mega menu |
| 主题目录 | `/directory` | 复用现有主题目录页 |
| 塔式 | `/` | 切换到塔式视图 |
| 纸片 | `/paper` | 切换到纸片视图 |
| 关于 | 暂无 | 第一版可禁用或仅展示 |
| 设置 | 暂无 | 第一版可禁用或仅展示 |

## 5. 推荐组件结构

### 5.1 保守改造方案

保留 `Header` 作为总入口，新建子组件承载菜单面板：

```text
src/components/ui/modules/Header/
├─ Header.jsx
├─ Header.module.scss
├─ HeaderMegaMenu.jsx
├─ HeaderMegaMenu.module.scss
├─ MobileHeaderMenu.jsx
└─ MobileHeaderMenu.module.scss
```

优点：

- 改动集中在 Header 模块内；
- 不增加新的全局模块边界；
- `Header.jsx` 保持调度角色；
- 桌面菜单和移动菜单样式隔离。

### 5.2 暂不推荐的新模块方案

不建议第一版新增：

```text
src/components/ui/modules/Navigation/
```

原因：

- 当前项目只有一个 Header 使用该菜单；
- 过早抽象会增加导出、文档和职责维护成本；
- Header 目录内聚更符合现有结构。

## 8. 动画方案

### 8.1 技术选择

第一版建议使用 CSS transition：

```text
opacity
transform
visibility
```

原因：

- Awwwards 菜单本身是面板展开，不需要复杂时间线；
- CSS 更容易配合响应式和 `prefers-reduced-motion`；
- 当前项目已有 `PageTransition` 使用 GSAP，Header 内不应过早增加 timeline 状态。

后续若要增强，可再将以下部分升级为 GSAP：

```text
面板 opacity + y
分组列表 stagger
右侧链接 stagger
遮罩淡入淡出
```

### 8.2 桌面端动画

打开：

```text
Overlay opacity: 0 -> 1
Panel opacity: 0 -> 1
Panel translateY: -12px -> 0
Panel scaleY: 0.98 -> 1
Links opacity: 0 -> 1
Links translateY: 8px -> 0
```

关闭：

```text
Panel opacity: 1 -> 0
Panel translateY: 0 -> -8px
Overlay opacity: 1 -> 0
```

### 8.3 移动端动画

打开：

```text
MobileMenu translateY: -12px -> 0
MobileMenu opacity: 0 -> 1
Rows opacity: 0 -> 1
```

关闭：

```text
MobileMenu opacity: 1 -> 0
```

### 8.4 动画约束

- 不使用 `sleep(1)` 或累计计时；
- 如使用 JS 动画，必须基于事件回调、CSS transition end 或 GSAP 完成回调；
- 若需要手写时间判断，必须基于 `time.time()` / `performance.now()` 的绝对时间差；
- `prefers-reduced-motion: reduce` 下直接显示最终态。

## 9. 状态与事件

`Header` 内部维护：

```text
isExploreOpen: boolean
isMobileMenuOpen: boolean
activeMegaSection: 'explore' | 'themes' | 'views' | 'project'
```

交互规则：

| 操作 | 结果 |
|------|------|
| 点击探索 | 打开或关闭桌面 mega menu |
| 点击汉堡 | 打开或关闭移动菜单 |
| 点击遮罩 | 关闭所有菜单 |
| 按 Esc | 关闭所有菜单 |
| 点击 `/directory` | 关闭菜单并调用 `onNavigate('/directory')` |
| 点击 `/` 或 `/paper` | 关闭菜单并调用对应路由 |
| 点击当前路由 | 只关闭菜单，不重复导航 |

背景滚动：

- 菜单打开时可给 `body` 添加锁滚动类；
- 当前项目滚动容器在 `AppLayout` 内，不只依赖 window；
- 第一版可以先不锁滚动，只用遮罩拦截点击；
- 若出现移动端背景滚动穿透，再补充 `body` 与滚动容器锁定。

## 10. 可访问性

必须满足：

- 菜单触发器使用 `button`；
- `Explore` 按钮设置 `aria-expanded`；
- 面板容器设置明确 `id` 并由按钮 `aria-controls` 指向；
- 当前路由使用 `aria-current="page"`；
- 移动端汉堡按钮设置 `aria-label`；
- Esc 可关闭；
- 焦点态清晰；
- 遮罩不应吞掉键盘用户的关闭能力。

第一版暂不强制做完整 focus trap，但需要保证：

```text
Tab 可以进入菜单项
Esc 可以退出
关闭后焦点回到触发按钮
```

## 14. 风险与约束

### 14.1 与 3D Canvas 层级冲突

建议层级：

```text
Scene Canvas: 1
Header: 1000
Header overlay: 1100
Header mega menu: 1200
ThemeDirectory page: 常规页面流
PageTransition wipe: 保持现有最高策略
```

实际以现有 SCSS 为准，不盲目把 z-index 提到过高。

### 14.2 Header 过重

如果 `Header.jsx` 超过合理复杂度，必须继续拆分：

```text
HeaderNav.jsx
HeaderActions.jsx
```

第一版只拆 `HeaderMegaMenu` 与 `MobileHeaderMenu`，保持 KISS。

### 14.3 搜索框误导

若第一版不做真实搜索，搜索框必须避免暗示可输入并查询。

建议：

```text
用 button 形式展示「探索主题」
或输入框 disabled，并明确 aria-disabled
```

### 14.4 与 Footer 模型切换重复

Header 中可以提供「塔式 / 纸片」入口，但 Footer 仍保留当前视图切换。

原因：

- Header 是全站导航；
- Footer 是当前 3D 视图的快捷切换；
- 两者职责不同，第一版允许重复入口。

后续如果用户体验显得重复，再评估是否移除 Footer。
